# 2026-04-16 安装包体积瘦身方案

> 状态：待评审
> 适用范围：Windows x64 安装包、GitHub Release tag 发布产物

## 目标

评估当前 exe 安装包是否还有继续瘦身的空间，并按收益、风险、实施成本给出分层方案，避免在低收益项上投入过多时间。

## 当前基线

基于 2026-04-16 本地产物与现有配置的观察，当前包体已经完成过一轮有效收缩：

- `app.asar` 已降到约 6.12 MB，说明业务代码本体已经较小。
- `win-unpacked` 总体积约 288.59 MB。
- GitHub Release 下载到的 setup.exe 约 83.8 MB。
- 当前已启用 `asar`。
- 当前已启用 `compression: maximum`。
- 当前 Electron 语言包已限制为 `zh-CN` 与 `en-US`。
- 当前已排除 `docs`、`scripts`、`dist-electron`、`build` 等明显不应入包目录。

这意味着当前下载包的主要体积来源已经不是业务代码，而是 Electron/Chromium 运行时本身。

## 方案分层

## A 类：低风险，收益较小，可立即做

这些项基本不会改动产品架构，但预估只能带来 0 MB 到 3 MB 级别的安装包下降。

### A1. 只保留 `zh-CN` 语言包

- 当前还保留了 `en-US`。
- 预估收益：安装包通常小于 0.5 MB。
- 风险：极低。
- 适合场景：明确只面向中文用户，且不考虑英文系统上的默认回退体验。
- 建议：可做，但优先级不高。

### A2. 继续收紧 `files` 白名单

- 目标是二次确认 `out/**` 之外没有意外生产依赖或静态资源被带入包体。
- 当前看 `app.asar` 已经很小，说明这里的收益不会太大。
- 预估收益：0 MB 到 1 MB。
- 风险：低。
- 建议：作为例行检查可做，但不应预期明显收益。

### A3. 检查图标、图片、内置静态资源是否还能压缩

- 如果安装包内还有高分辨率 PNG、未压缩模板、历史遗留资源，可以继续裁剪。
- 当前从结果看，这类资源不太可能是主要瓶颈。
- 预估收益：0 MB 到 1 MB。
- 风险：低。
- 建议：可顺手做，不建议单独立项。

## B 类：中风险，可实验，收益不确定

这些项理论上可能继续省几 MB，但兼容性风险明显升高，需要专门验证安装、启动、渲染、更新链路。

### B1. 评估裁剪图形相关运行库

候选文件通常包括：

- `dxcompiler.dll`
- `dxil.dll`
- `vulkan-1.dll`
- `vk_swiftshader.dll`
- `libEGL.dll`
- `libGLESv2.dll`

说明：

- 这些文件与 GPU 渲染、软件回退渲染、某些图形后端能力相关。
- 某些极简 Electron 应用在特定场景可以去掉一部分，但稳定性风险高。
- 可能出现启动黑屏、渲染异常、兼容性下降、部分机器无法启动等问题。

预估收益：

- 如果裁掉一部分，理论上可能节省 5 MB 到 30 MB 未压缩体积。
- 但最终折算到安装包，通常不会线性等比下降。

建议：

- 不建议直接在正式分支裁。
- 如果要试，只能做单独分支实验，并覆盖多机型验证。

### B2. 评估切换目标发布形态

候选形态：

- NSIS 安装包
- Portable 绿色版
- 压缩包分发

说明：

- 不同分发形态的下载体积会有一些差异。
- 但通常不是数量级变化。
- 同时会影响安装体验、桌面快捷方式、卸载流程和自动更新策略。

预估收益：

- 通常是小幅变化。

建议：

- 如果产品目标强调“下载安装即用”，继续保留 NSIS 更稳妥。
- 不建议仅为了几 MB 改变主发包形态。

### B3. 跟随 Electron 大版本观察包体变化

说明：

- 不同 Electron 版本对 Chromium runtime、编译参数、依赖组件的体积影响并不完全一致。
- 某些版本会略增，某些版本会略降。

预估收益：

- 不确定，可能更小，也可能更大。

建议：

- 作为升级附带观察项，不要把它当成主要瘦身手段。

## C 类：高成本，但只有这类方案才可能明显降体积

如果目标不是“再降几 MB”，而是希望从 83 MB 继续打到 40 MB、20 MB 甚至更小，就不能继续只盯 Electron Builder 配置，需要改架构。

### C1. 从 Electron 迁移到 Tauri

- 使用系统 WebView 替代内置 Chromium。
- 对下载包体积通常有明显帮助。
- 代价是主进程、IPC、自动更新、打包链路需要整体重构。
- 与现有 Electron 生态代码不能无缝平移。

建议：

- 只有在“安装包体积”被提升到产品关键指标时才值得评估。

### C2. 从 Electron 迁移到 Wails 或其他系统 WebView 路线

- 原理与 Tauri 相近，重点也是不再把 Chromium 整包打进去。
- 代价同样是架构迁移和工程重建。

建议：

- 与 Tauri 一起做路线对比，不要并行推进。

### C3. 调整产品形态，改成更薄的本地壳

- 将更多能力迁移到服务端或在线工作流。
- 本地桌面端只保留输入、预览、登录和最小控制面。
- 这样不仅可能减小包体，也可能降低桌面端更新压力。

代价：

- 需要重新设计权限边界、离线能力、服务端成本和隐私边界。

## 建议优先级

如果目标是“当前版本继续做一点合理瘦身，但不折腾架构”，建议按下面顺序推进：

1. 保持现有 NSIS + asar + maximum 压缩策略不变。
2. 视产品定位决定是否去掉 `en-US`。
3. 再做一次 `files` 白名单与静态资源清点，确认没有遗漏的无用文件。
4. 到此为止，如果安装包只下降了 1 MB 到 3 MB，应视为正常结果，不再继续深挖。

如果目标是“显著降低安装包体积”，建议直接转为路线评估：

1. 定义目标值，例如 setup.exe 是否必须低于 50 MB。
2. 如果目标明确低于 50 MB，则优先评估 Tauri 或系统 WebView 路线。
3. 不建议在当前 Electron 方案上投入大量时间做高风险 DLL 裁剪。

## 推荐结论

基于当前现状，推荐结论如下：

- 当前 83.8 MB 的 Windows 安装包体积处于合理区间。
- 当前配置层面的主要瘦身工作已经基本做完。
- 继续优化仍有空间，但多半只剩小幅下降，不会再有数量级收益。
- 如果后续没有“必须明显降体积”的硬性指标，建议把精力转回产品功能、稳定性和发布链路验证。
- 如果后续将“包体降到某个明确阈值”列为正式目标，应直接进入桌面技术栈替代评估，而不是继续在 Electron Builder 参数层面微调。
